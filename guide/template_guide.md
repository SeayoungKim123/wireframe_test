# 화면 설계 템플릿 — 사용 가이드

*작성일: 2026-04-18 / 최종 수정: 2026-05-01*

**독자**: 이 템플릿으로 기획서를 작성하는 PM.

---

## 1. 이게 뭐예요?

HTML 파일 **하나**에 와이어프레임 + 핀 설명 + 버전 기록을 모두 담은 기획서 템플릿.

**언제 쓰면 좋나**
- 서버·툴 없이 파일 하나로 공유하고 싶을 때
- Confluence / Notion 업로드 전 초안을 빠르게 만들 때
- 변경 이력을 한 파일 안에 남기고 싶을 때

**언제 안 맞나**
- 여러 사람이 **동시에** 편집해야 할 때 (실시간 협업 불가)
- 복잡한 인터랙티브 프로토타입 (단순 와이어프레임 중심)

---

## 2. 시작하기 (3단계, 1분)

### Step 1 — 폴더 복사
이 repo의 **`template/` 폴더를 통째로 복사**해서 새 위치에 두세요. 폴더 안에는 셋이 함께 들어있습니다:

- `spec_template_base.html` — 본체 (이름만 `reading_log.html` 식으로 변경)
- `shell.css` — 공용 스타일
- `shell.js` — 공용 스크립트

> ⚠ HTML이 같은 폴더의 `shell.css` / `shell.js`를 `<link>` / `<script src>` 로 참조합니다. 셋이 같은 폴더에 있어야 정상 동작합니다.

### Step 2 — 설정 2줄 수정
에디터로 열고 `@ANCHOR:CONFIG` 검색. 블록 안 2줄만 고침:

```js
const TEMPLATE_CONFIG = {
  PAGE_TITLE:  '독서 기록표',          // 상단에 표시될 이름
  STORAGE_KEY: 'spec_reading_log',   // 기획서마다 고유 (영문+숫자)
};
```

> ⚠ `STORAGE_KEY`는 **기획서마다 반드시 달라야** 함. 같으면 서로 편집 내용이 섞임.

### Step 3 — 브라우저로 열기
파일을 더블클릭. 별도 서버 불필요.

---

## 3. 와이어프레임 그리기

`@ANCHOR:CONTENT:START` 와 `@ANCHOR:CONTENT:END` **사이에만** HTML을 씁니다.

**핀을 달 수 있게 하려면**: 요소에 `data-component="고유이름"` 추가.

```html
<div data-component="login-form">
  <input placeholder="이메일">
  <button>로그인</button>
</div>
```

비유: `data-component`는 컴포넌트의 **이름표**. 이름 없는 요소엔 핀을 못 붙입니다.

### Claude에게 요청하는 방법

> "`reading_log.html`에 회원가입 폼 추가해줘. `data-component="signup-form"` 붙여서."

Claude는 `@ANCHOR:CONTENT` 영역만 수정하면 됩니다. 공용 인프라(CSS·JS)는 `shell.css` / `shell.js` 로 외부화되어 있어 HTML 본체는 ~270줄 수준 → 토큰·맥락 부담 최소.

## 4. 핀 달기 (브라우저에서)

### Step 1 — 편집 모드 진입
URL 끝에 `?edit=1` 추가.
예: `file:///.../reading_log.html?edit=1`
→ 상단바에 "편집 모드" 토글이 나타남.

### Step 2 — 토글 켜기
1. **번호 표시 토글** — 우측 패널 상단의 토글이 디폴트로 ON 상태이며, 점선 테두리로 핀 달 수 있는 요소가 표시됨 (필요 시 OFF로 전환하면 "번호 숨김"으로 라벨이 바뀜)
2. **편집 모드 토글 ON** — 상단에 "편집 중" 뱃지, 하단에 편집 툴바 등장

### Step 3 — 핀 배치
1. 하단 툴바의 **`+ 새 핀`** 클릭
2. 커서가 십자 모양으로 변함 → 핀 달 컴포넌트 **클릭**
3. 클릭 위치에 보라색 핀 생성, 우측 **컴포넌트 설명 패널**의 해당 항목이 펼쳐지면서 강조됨

### Step 4 — 패널에서 설명 채우기

우측 패널은 **항상 노출**되어 있고, 핀별로 **아코디언 항목** 한 개씩 쌓입니다.

- 헤더(번호 + 제목) 클릭으로 펼침/접기 (여러 항목 동시 펼침 가능, 기본은 모두 접힘)
- 와이어프레임의 **핀을 클릭**하면 해당 항목이 자동 펼침 + 패널에서 그 위치로 스크롤 + 헤더 강조
- 펼친 상태에서 **제목 / 역할 / 상태 / 속성 / Confluence 링크** — 클릭해서 직접 타이핑
- 핀을 드래그해서 위치 조정 가능
- 항목 헤더 우측 **🗑 버튼**으로 핀 삭제 (편집 모드일 때만 표시)

---

## 5. 저장하기 (3단계)

저장 방식이 3개인 이유: **편집 안전장치**와 **영구 기록**을 분리하려고.

비유: 포스트잇 → 서랍 → 공식 도장.

| 단계 | 언제 | 결과 |
|---|---|---|
| **자동 저장** | 편집할 때마다 자동 | 브라우저 안에만 (포스트잇) |
| **💾 작업본 저장** | 퇴근 / 잠시 공유 | HTML 파일 다운로드 (서랍) |
| **📌 버전 저장** | 리뷰 완료 · 마일스톤 | HTML 다운로드 + 새 버전 id 부여 (공식 도장) |

### ⚠ 자동 저장은 날아갈 수 있습니다

- 브라우저 데이터 초기화
- 시크릿 / 프라이빗 모드
- 다른 PC · 브라우저에서 열기
- 파일 이름·경로 변경

→ 중요한 작업은 반드시 **💾 또는 📌**로 저장.

### 📌 버전 저장 시 선택

| 유형 | 언제 | 예시 |
|---|---|---|
| 마이너 (v0.2 → v0.3) | 문구 수정, 섹션 추가, 피드백 반영 | "로그인 에러 메시지 추가" |
| 메이저 (v0.5 → v1.0) | 공식 리뷰 통과, 방향성 재설계, 외부 공유 | "최종본" |

## 6. 버전 확인 / 비교

### 과거 버전 열기

상단바 **버전 드롭다운** 클릭 → 버전 선택.

- **단일 파일 모델** (작업 중인 한 파일에 versions[] 누적): 같은 페이지 안에서 표시 전환. 과거 버전은 회색 "읽기 전용" 배너 + 편집 버튼 비활성.
- **분리 모델** (외부 공유용 — §10 참조): 드롭다운에 **형제 버전 파일들**이 노출됨. 다른 버전 클릭 시 해당 파일로 이동.

### 두 버전 비교 (직전 버전 대비)

1. 상단바 **🔀 비교** 토글 ON
2. 핀이 색깔로 구분됨:
   - 🟢 초록 — 추가됨
   - 🟠 주황 — 수정됨
   - ⚪ 회색 고스트 — 삭제됨
3. 패널의 해당 핀 항목 헤더에 색상 사이드바 + 변경 요약 띠 (`✏️ 변경: 역할, 상태`)
4. 펼치면 바뀐 섹션(역할/상태/속성/Confluence)에 노란 배경

> 분리 모델에서 비교 모드는 **직전 버전 파일을 fetch**해서 baseline으로 사용합니다. 따라서 **로컬 서버**나 **vercel 배포본**에서만 동작 (§10 로컬 서버 안내). 자기 자신이 첫 버전(예: v0.1)이거나 fetch 실패 시 토글은 비활성.

## 7. Claude에게 수정 요청하기

앵커를 활용하면 토큰이 확 줄어듭니다.

### 내용 수정 — CONTENT 영역 명시
> "`reading_log.html`의 `@ANCHOR:CONTENT` 영역에 검색창 추가해줘."

### 설정만 바꾸기 — CONFIG 명시
> "페이지 타이틀을 '독서 다이어리'로 바꿔줘."

### 핀 데이터는 원칙적으로 Claude에게 맡기지 않기

핀 데이터(`@ANCHOR:DATA`)는 **브라우저가 자동 관리**. Claude가 직접 고치면 구조가 깨질 수 있음 → **편집 모드에서 직접 수정**.

**예외**: 초안 핀을 대량 생성할 때만 Claude에게 `spec-data` JSON 작성 요청 가능.

## 8. 자주 묻는 질문

**Q. 파일 이름을 바꿔도 되나요?**
됩니다. 단, 자동 저장(localStorage)은 파일 경로 기준이라 사라질 수 있습니다. 💾/📌로 저장된 건 파일 안에 있으니 안전.

**Q. 다른 PC에서 열었더니 편집 내용이 없어요.**
자동 저장은 해당 브라우저 로컬에만 있습니다. 💾 작업본 저장한 파일을 가져오세요.

**Q. 편집 버튼이 안 보여요.**
URL 끝에 `?edit=1` 있는지 확인. 읽기용 공유 시엔 일부러 빼면 됩니다.

**Q. 핀이 엉뚱한 곳에 있어요.**
편집 모드에서 핀을 드래그하세요. 위치는 컴포넌트 내 상대좌표라 창 크기가 바뀌어도 유지됩니다.

**Q. "그 사이 문서가 바뀌었어요" 모달이 떴어요.**
내 편집 기준 버전보다 HTML에 더 최신 버전이 있다는 뜻. 보통 Claude가 그 사이 파일을 수정했을 때. 선택지:
- **내 편집 버리기** — 최신본 위에서 다시 시작
- **내 편집을 새 버전으로** — 내 변경을 새 버전으로 커밋

**Q. 전부 되돌리고 처음부터 하고 싶어요.**
편집 툴바의 **↺ 초기화**. 자동 저장(draft)만 지움 — 저장된 버전엔 영향 없음.

**Q. 한 파일에 여러 화면 만들어도 되나요?**
권장하지 않음. **화면 1개 = 파일 1개** 원칙. 여러 화면은 파일을 나누고 `STORAGE_KEY`를 각각 다르게.

**Q. 버전 드롭다운에 자기 버전만 보여요.**
분리 모델(§10)에서 `manifest.json` fetch가 실패한 경우. 보통 파일을 **더블클릭(`file://`)으로 직접 연** 상태입니다. 로컬 서버를 띄우고 `http://localhost/...`로 접속하세요(§10).

**Q. 🔀 비교 토글이 비활성이에요.**
- 자기 페이지가 **첫 버전**(예: v0.1)이라 직전 버전이 없을 때 → 정상.
- 분리 모델에서 manifest 또는 직전 버전 파일 fetch 실패 시 → 로컬 서버 사용 권장(§10).

**Q. 핀 위에서 숫자 클릭과 핀 헤더 토글이 헷갈려요.**
와이어프레임의 핀 클릭 = 우측 패널 해당 항목 자동 펼침 + 강조. 패널 안 헤더 클릭 = 그 항목만 펼침/접기. 두 동선이 같은 결과로 수렴하니 어느 쪽이든 OK.

## 9. 외부 공유 (Vercel 배포 — 분리 모델)

### 왜 분리 모델로 가나?

단일 파일 + versions[] 누적 모델은 작업·공유엔 편하지만, 외부에 **버전 진화 카탈로그**로 공유할 땐 헷갈립니다. 콘텐츠 HTML은 한 벌이라 v0.1을 보고 있어도 v0.2 와이어프레임이 보이거든요 (핀 설명만 v0.1 시점).

이때 **파일 분리 + 정적 인덱스 카탈로그** 모델로 옮깁니다. Vercel 정적 호스팅과 매우 잘 맞습니다.

### 폴더 구조

```
sample/<기획서명>/
├── index.html           ← 카탈로그 페이지 (버전 카드 리스트, JS 없음)
├── shell.css            ← 공용 카메라 (CSS) — 모든 버전이 공유
├── shell.js             ← 공용 카메라 (JS)  — 모든 버전이 공유
└── versions/
    ├── manifest.json    ← 형제 버전 목록 (단일 진실 소스)
    ├── v0.1.html        ← 자기 버전 1개만 보유, ../shell.css/.js 참조
    ├── v0.2.html
    └── ...
```

각 버전 파일은:
- `versions[]`에 자기 버전 1개만 (다른 버전 정보는 manifest로 관리)
- `STORAGE_KEY`는 파일별로 분리 (예: `spec_reading_log_v0_1`, `spec_reading_log_v0_2`) → localStorage 충돌 방지
- `PAGE_TITLE`에 버전 식별 (예: `독서 기록표 v0.2`)
- `<link href="../shell.css">` 와 `<script src="../shell.js">` 로 한 단계 위 폴더의 셸 파일 참조

### `manifest.json` 형식

```json
{
  "versions": [
    { "id": "v0.1", "file": "v0.1.html", "memo": "...", "timestamp": "...", "pinCount": 7 },
    { "id": "v0.2", "file": "v0.2.html", "memo": "...", "timestamp": "...", "pinCount": 8 }
  ],
  "current": "v0.2"
}
```

페이지가 로드되면 manifest를 fetch해서:
- 상단 **버전 드롭다운**에 형제 버전 노출 → 클릭 시 해당 파일로 이동
- **🔀 비교 모드**의 baseline으로 직전 버전 파일을 추가 fetch → specs 추출 → 정책대로 핀 추가/수정/삭제 강조

### 새 버전 추가 흐름 (운영)

1. 가장 최근 버전 파일을 `?edit=1`로 열고 편집
2. **📌 버전 저장**으로 다음 버전 HTML 다운로드 → `versions/v0.X.html`로 이름 변경
3. (필요 시) 새 파일의 `versions[]`를 자기 버전 1개로 정리하고 `STORAGE_KEY`/`PAGE_TITLE` 갱신
4. `versions/manifest.json` 에 새 항목 추가 (id / file / memo / timestamp / pinCount)
5. `index.html` 에 카드 한 개 끼워 넣기 (최신을 맨 위)
6. `git commit && git push` → vercel 자동 재배포

> 다운로드된 HTML은 **상대경로로 `../shell.css` / `../shell.js`를 참조**합니다. `versions/` 폴더 안에 두는 한 자동으로 동작합니다 (셸 파일은 한 번 셋업 후 손댈 일 없음).

### ⚠ 로컬 작업 시 서버 필요

`manifest.json` fetch와 직전 버전 fetch는 `file://` 프로토콜에서 막힙니다. 로컬에서 미리 보려면 **로컬 서버**를 띄우세요:

| 도구 | 명령 | 비고 |
|---|---|---|
| VS Code Live Server 확장 | (확장 설치 후) `index.html` 우클릭 → Open with Live Server | GUI, PM 친화 |
| http-server | `npx http-server sample -p 5173` | Node.js 필요 |
| Vercel dev | `npx vercel dev` | 배포 환경과 동일 |

그러고 `http://localhost:.../reading_log/`로 접속. **공유받는 분들은 Vercel URL(https://)로 보면 자동으로 정상.**

### 배포본은 "읽기 전용"

Vercel에 올라간 파일에서 `?edit=1`로 편집해도 본인 브라우저 localStorage에만 남고 다른 사람에게 반영 안 됩니다. **편집은 항상 로컬에서 → git push → 자동 재배포** 흐름으로 운영하세요.
